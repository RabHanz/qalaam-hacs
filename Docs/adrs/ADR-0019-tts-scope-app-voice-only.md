# ADR-0019: TTS scope — app-voice only; Quranic verses are NEVER synthesized by general-purpose TTS

- **Status:** Accepted
- **Date:** 2026-05-04
- **Deciders:** Qalaam maintainers
- **Outcome served:** O-06, O-08, O-13 (foundation; voice-quality + adab guard)
- **Consulted:** ADR-0006 (Habibi-TTS-MSA), ADR-0007 (Qalaam-house voice cap)
- **Informed:** all engineering, all product

## Context

ElevenLabs (and any other general-purpose Arabic TTS available in v0.1) is
trained on conversational + narrative MSA. It does NOT model the rules of
tajweed: madd lengths (2/4/6 harakat), ghunnah duration, qalqalah, ikhfa,
idgham, hamzat al-wasl elision, sukoon-rules, qira'at variations. It will
mispronounce hamzat al-wasl, miss madd, get qalqalah wrong, and produce a
flat MSA reading that any hifdh student will immediately call wrong.

Beyond the technical inadequacy, a synthesized Quranic verse violates the
spirit of سَمَاع — the Quran is heard from qualified reciters with isnad,
not generated by a model. Even when technically passable, AI-rendered
Qur'an sets a precedent we are not willing to enable in v0.x.

The earlier slug naming (`qalaam-house` / `qalaam-house-mujawwad`) blurred
the line between "app voice" and "house recitation voice" because the same
provider handle could in principle serve both. That ambiguity is itself a
risk: a future bug in the orchestration layer could route a verse through
ElevenLabs without anyone noticing until users heard it.

## Decision

We will split the TTS surface into two non-overlapping voice categories,
enforce the boundary BOTH in naming AND in a runtime guard, and ship the
guard before any production traffic reaches the TTS worker:

1. **App-voice slugs** (`qalaam-app-voice`, `qalaam-app-voice-warm`) serve
   only Qalaam UI/system speech: settings prompts, deep-study explanations,
   session cues, daily summary narration, page-turn cues, app onboarding.
   Routed to ElevenLabs in v0.1 (per ADR-0014). Never used for verses.
2. **Recitation slug** (`qalaam-house-mujawwad`) is RESERVED for
   Habibi-TTS-MSA fine-tuned on the Quran corpus (per ADR-0006). General
   TTS providers refuse it at synthesize time. Becomes available in v2.5+
   when the Habibi backend lands and reciter licensing per ADR-0007 is
   resolved.

The runtime guard (`services/tts-worker/.../quranic_guard.py`) refuses any
synthesize request that:
- Targets a recitation-reserved slug.
- Carries a `verse_key` field (always Quranic by definition).
- Has text containing the verse-end glyph U+06DD.
- Has text containing hamzat al-wasl alif U+0671 (Uthmani-script signal).
- Has tashkeel density > 25% on Arabic letters of length ≥ 8.
- Begins with a known Quranic opener fingerprint.

Refusals are surfaced as HTTP 422 with a structured hint pointing at
`/v1/audio/by_verse/<verse_key>/<reciter>` so callers can self-correct.

## Alternatives considered

1. **Status quo** — keep the `qalaam-house` umbrella slug and trust the
   orchestration layer to never feed verses to it. **Rejected because** the
   trust is implicit; one bad PR or one rushed feature could route a verse
   through. Defense in depth required.
2. **Document the boundary in CLAUDE.md only** (no runtime guard). **Rejected
   because** prose policy is ineffective when the cost of regression
   (synthesizing Qur'an) is irreversible reputational harm.
3. **Use ElevenLabs voice-conditioning to "teach" tajweed via prompt
   engineering.** **Rejected because** ElevenLabs has no acoustic model of
   tajweed rules; the output would be dressed-up MSA, not recitation. False
   confidence is worse than a clear refusal.
4. **Block the entire TTS provider until Habibi is ready.** **Rejected
   because** the app-speech use case (UI narration) is genuinely useful and
   has no halal-vs-haram dimension; pulling it would punish users for an
   architectural concern they shouldn't see.

## Consequences

### Positive

- Boundary is now enforceable in code, not only in policy.
- 422 + structured hint makes the misroute self-fixing for callers.
- Clear semantic split (`intended_use: "app-speech"` vs `"recitation"`) makes
  the voices catalog self-documenting in `/v1/voices`.
- Future TTS providers (CosyVoice 2 streaming, etc.) inherit the same guard
  by importing one module — no per-provider re-enforcement needed.

### Negative

- Slug rename is a breaking change for any caller still using
  `qalaam-house`. Caught at request-validation time (Pydantic Literal); 422
  with clear error.
- Conservative detector may false-positive on heavily-tashkeel'd MSA poetry.
  Acceptable: that copy is ~0% of Qalaam's TTS load, and the workaround
  (route through a different provider) is documented in the 422 body.

### Neutral

- The `qalaam-house-mujawwad` slug is advertised in `/v1/voices` so callers
  can see the future surface; trying to use it now returns 422.

## Risks & monitoring

- **Risk:** detector misses a Quranic-text shape we didn't anticipate
  (e.g., a hadith quote that includes a verse fragment without hamzat
  al-wasl). **Leading indicator:** TTS worker `tts.synthesize.failed` logs
  containing Arabic-only text — sample weekly. **Mitigation:** add a manual
  review gate when `arabic_chars_ratio > 0.95` and the text isn't refused
  (false-negative trap).
- **Risk:** future provider added without importing the guard. **Leading
  indicator:** new provider class lands without `refuse_if_quranic(req)` at
  the top of `synthesize`. **Mitigation:** lint/codegen rule that every
  TTSProvider's `synthesize` must call the guard.
